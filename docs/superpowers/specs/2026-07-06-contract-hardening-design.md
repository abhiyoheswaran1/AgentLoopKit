# Contract Hardening (`agentloop harden`) — Design

- Date: 2026-07-06
- Status: approved for planning
- Scope: Spec 1 of the front-of-loop suite (engine + flagship command). Follow-on specs (architecture-improvement interrogation, spec/PRD synthesis, plan→issues) are out of scope here and reuse this engine.

## Problem Statement

AgentLoopKit is strong on the back half of the loop — verification evidence, review-readiness, handoffs. The front half is thin. A task contract today is mostly captured intent: `create-task` scaffolds the template and `findPlaceholderTaskSections` flags empty sections, but nothing pressure-tests the contract for underspecified scope, untestable acceptance criteria, unstated assumptions, or internal contradictions.

The cost lands late: the review gate already lists "Ambiguous acceptance criteria" and "Conflicting requirements" as escalation triggers. The loop *detects* these weaknesses at review time and does nothing to resolve them earlier. Misalignment that could have been closed in five minutes at intake instead surfaces after a diff exists.

## Desired Outcome

A front-of-loop pass, `agentloop harden`, that interrogates a draft task contract until every blocking decision point ("soft spot") is resolved, writes the resolutions back into the contract, and emits gate-checkable assertions so the diff is later verified against what was agreed. Hardening turns front-of-loop alignment into an auditable artifact, not just a conversation.

## Non-Goals

- No LLM wrapper, no network calls, no API keys, no telemetry. The agent conducts the interrogation; AgentLoopKit provides structure, detection, and capture. (Enforces ROADMAP Non-Goals.)
- No new hosted service, database, or IDE surface.
- Not the follow-on front-of-loop capabilities (architecture interrogation, PRD synthesis, plan→issues). Those are separate specs consuming this engine.
- Does not conduct the questioning itself deterministically (rejected Approach B) — the engine reports soft spots; the agent asks the human.

## Architecture

Agent-conducted, AgentLoopKit-scaffolded. Three deterministic, LLM-free pieces:

1. **The engine** (`src/core/harden.ts`) — pure analysis. Given a parsed task contract, return a list of open **soft spots**. No model, no I/O beyond reading the contract. Snapshot-testable, fits the contract-lock discipline.
2. **The playbook** (`.agentloop/harden-playbook.md`, generated by `init`) — agent-facing instructions for *how* to interrogate, in the style of `.agentloop/agents/*.md`.
3. **The commands** (`src/cli/commands/harden.ts`) — `agentloop harden` (report soft spots; exit non-zero while blocking spots remain) and `agentloop harden --resolve` (write answers back + record the Hardening Log + emit gate assertions).

### Soft-spot taxonomy

Each soft-spot type maps onto an existing task-contract section. Detection is heuristic and deterministic.

| Type | Detected from | Severity | Example open question |
|---|---|---|---|
| `placeholder` | extends `findPlaceholderTaskSections` (empty/template-default section) | blocking | "Non-Goals is empty — what is explicitly out of scope?" |
| `untestable-acceptance` | Acceptance Criteria line with no verifiable predicate (no command, output, file, or measurable condition) | blocking | "'Works well' isn't checkable — what command or output proves it?" |
| `unbounded-scope` | Likely Files empty/overly broad, or Files-Not-To-Touch empty | blocking | "Which files or areas must this change *not* touch?" |
| `unstated-assumption` | Assumptions section thin relative to task type | advisory | "This is a `feature` task with no assumptions listed — what are you taking for granted?" |
| `contradiction` | token/area overlap between Acceptance Criteria and Non-Goals / Constraints | blocking | "Acceptance implies auth changes but Non-Goals excludes auth — which wins?" |

Detection rules live beside the taxonomy and are individually unit-tested. Thresholds (e.g. "thin relative to task type") are constants, tunable without touching call sites.

### Soft-spot data shape

```
SoftSpot {
  id: string          // stable, derived from type + section + ordinal
  type: SoftSpotType
  section: string     // contract section it belongs to
  question: string    // the open question the agent must resolve
  severity: 'blocking' | 'advisory'
  status: 'open' | 'resolved'
}
```

`id` is deterministic (no timestamps/random) so `--json` output is snapshot-stable and `--resolve <id>` is addressable across runs.

## The playbook

Generated into `.agentloop/harden-playbook.md` by `init` (added to the same generation path as the agent roster and gates). Contents:

- **The loop:** read open soft spots → interrogate the human one question at a time → resolve → re-run until no blocking spots remain (mirrors the loop-until-dry convergence used elsewhere).
- **The stop condition:** zero blocking soft spots. Advisory spots may remain, recorded but not blocking.
- **The stance:** interrogation, not agreement. Push back on vague answers; a resolved soft spot must be concrete enough to write into the contract and check a diff against.

## Commands

### `agentloop harden [task]`

- Resolves the target contract (active task if omitted, same resolution as other task commands).
- Runs the engine, prints open soft spots grouped by severity, with the questions.
- Exit code: non-zero while any **blocking** soft spot is open; zero otherwise. This makes `harden` directly usable as a CI/gate check.
- `--json`: emits the soft-spot list in a contract-locked shape (deterministic ordering, CI-env-independent, consistent with the existing `--json` contract discipline).

### `agentloop harden --resolve`

- Accepts resolutions as `--resolve <id> --answer "..."` (repeatable) or `--resolve-file <path>` for batch.
- For each resolution: writes the answer into the correct contract section, and appends an entry to a `## Hardening Log` block in the contract (soft-spot id, resolution text, contract-relative provenance — no wall-clock timestamp, to stay deterministic for tests; ordering is by resolution sequence).
- Re-runs the engine after writing so the caller sees remaining spots.

### Gate-assertion emission

Resolutions that constrain the diff become assertions the existing gates check the actual diff against:

- Resolved `unbounded-scope` / Files-Not-To-Touch → the review and security gates flag a diff that touches an excluded area.
- Resolved Non-Goals → same, as a scope-creep signal.

Assertions are recorded in a structured block the gate logic reads (reusing the existing gate/guard evaluation path — no new gate engine). This is the forward-wiring that makes hardening pay off later: "you said auth was out of scope → the diff touches `auth/` → the gate flags it."

## Integration points

- **`create-task`** gains an optional `--harden` flag: after scaffolding, it runs the engine and instructs the agent (via generated loop docs) to harden the contract before marking it ready. Default behavior unchanged when the flag is absent.
- **`check-gates` / `ready`**: open blocking soft spots **warn by default and fail under `--strict`** — consistent with the existing `doctor --advisory/--strict` and `dogfood`/`dogfood:strict` duality. `agentloop harden` itself is the hard gate for teams that want to block directly.
- All of the above is **additive and opt-in** — no change to existing command behavior, preserving the 1.0 stability contract.

## Error handling

- Missing/malformed contract → existing `AgentLoopError` / `TaskOutputPathError` patterns.
- `--resolve <id>` for an unknown/already-resolved id → clear `AgentLoopError`, non-zero exit, no partial write.
- Batch `--resolve-file` is all-or-nothing per file parse; a malformed file fails before any contract mutation.

## Testing

- **Engine unit tests** on fixture contracts: a deliberately vague fixture → expected N soft spots by type/severity; the committed 1.0-stability contract → zero blocking soft spots (doubles as a regression anchor proving the detector doesn't false-positive on a known-good contract).
- **Detection-rule tests**: one focused test per soft-spot type, including boundary cases for each heuristic threshold.
- **`--resolve` tests**: answer lands in the correct section; Hardening Log entry is written; re-run reflects the resolved state; unknown-id path errors cleanly.
- **Contract-lock**: `harden` CLI help, `--json` shape, and any new config surface get contract-lock snapshots like every stable command; `contract:check` stays green.
- **Gate-assertion tests**: a resolved Files-Not-To-Touch assertion + a diff touching that area → gate flags it (warn) and fails under `--strict`.

## Rollback

Fully additive: new engine module, new command, new generated playbook, new optional flags, new tests. Revert by dropping `src/core/harden.ts`, `src/cli/commands/harden.ts`, the playbook generation, the `--harden`/gate wiring, and the associated tests. No existing runtime behavior is removed.

## Open follow-ons (separate specs)

- Architecture-improvement interrogation (scan → present → harden selections).
- Spec/PRD synthesis from a hardened contract.
- Plan → independently-grabbable issues.

Each is a thin consumer of the `harden` engine and gets its own spec → plan → implementation cycle.
