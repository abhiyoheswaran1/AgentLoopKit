# AgentLoopKit 1.0 Consistency Audit

This is a point-in-time sweep of the ~39-command stable surface
(`STABLE_COMMANDS` in `src/core/stable-surface.ts`) ahead of the 1.0 freeze.
Every finding below cites the exact source location and, where practical, a
reproduced command/output pair. Each finding carries exactly one triage tag:

- **fix-before-1.0** ‚Äî fixing this after the freeze would require breaking an
  already-committed name, shape, or behavior (a 2.0 event). Fix now while it's
  still free.
- **accept-as-is** ‚Äî fine to freeze; either genuinely consistent, or a gap
  that can be closed additively in a later 1.x release without breaking
  anything.
- **defer-to-experimental** ‚Äî the surface is not mature enough to freeze at
  all; reclassify it as experimental (per `docs/versioning.md`, Task 8)
  instead of committing to it.

## Summary

| Tag | Count |
| --- | --- |
| fix-before-1.0 | 8 |
| accept-as-is | 9 |
| defer-to-experimental | 0 |

No surface was judged too immature to freeze outright (no
**defer-to-experimental** findings). The command surface is largely
consistent ‚Äî most of the ~39 commands share the same option names, error
envelope, and exit-code model. The fix-before-1.0 findings below are
concentrated in three areas: one genuinely overloaded flag name
(`--task`), a handful of commands that never got wired into the shared JSON
error/redaction helpers, and JSON-envelope field names that drifted between
commands built at different times (`status` vs `overallStatus`, `nextAction`
vs `nextCommand` vs bare `command`/`reason` vs `nextActions`). None of the
fixes are large; all are isolated to 1-3 files each, matching the "variable
size, cheap per finding" shape Task 7b expects.

## Method

- Read every file in `src/cli/commands/*.ts` (39 files) that backs a
  `STABLE_COMMANDS` entry, plus the `src/core/*.ts` modules they call into.
- Grepped for every `.option(...)` registration of `--redact-paths`,
  `--json`, `--strict`, `--task` across all command files and diffed the
  descriptions/semantics.
- Cross-referenced the `--json` output shapes against the locked snapshots in
  `tests/contract/__snapshots__/json-shape.contract.test.ts.snap` (the only
  8 commands currently under contract lock: `status`, `next`, `start`,
  `ready`, `guard`, `check-gates`, `review-context`, `artifacts`).
- Grepped every `process.exitCode` / `process.exit` assignment in
  `src/cli/commands/*.ts` and traced each back to the status enum that
  triggers it.
- Built and ran the compiled CLI (`node dist/cli/index.js ...`) to confirm
  several findings live rather than relying on static reading alone (exit
  codes, `--help` text, and one JSON-error-envelope reproduction).

## 2. JSON envelope

**Error shape is consistent** where it's wired up: every command that
handles its own JSON errors funnels through the same
`{ error: { code, message, ...details } }` shape (`src/cli/json-errors.ts:17-32`,
duplicated locally with an identical shape in `badge.ts`, `create-task.ts`,
`task.ts`, `policy.ts`, `npm-status.ts`, `release-proof.ts`, etc.). This is
evidence the surface is *not* randomly inconsistent ‚Äî it's a copy-pasted
convention that happens to have one hole (next finding).

- **`github import --json` does not emit JSON on error** ‚Äî `src/cli/commands/github.ts`
  has no try/catch around `importGithubMetadata(...)`, so any
  `AgentLoopError` it throws (missing/invalid `--issue-json`, path escapes
  the repo, etc. ‚Äî see `src/core/github-metadata.ts:229-289`) falls through to
  the generic top-level handler in `src/cli/index.ts:91-95`, which prints
  plain text to stderr, not JSON. Reproduced live:

  ```
  $ node dist/cli/index.js github import --json --issue-json /tmp/does-not-exist-xyz.json
  agentloop: GitHub metadata JSON path must stay inside the repository: /tmp/does-not-exist-xyz.json
  EXIT=1
  ```

  Every other JSON-capable stable command guarantees a JSON envelope on
  error too; a caller scripting against `agentloop github import --json`
  and piping to `JSON.parse` will crash. **fix-before-1.0**.

- **The "overall status" field name is split 50/50 between `status` and
  `overallStatus`** for the same concept (an aggregate pass/warn/fail
  verdict): `src/core/ready.ts:46` (`status: ReadyStatus`),
  `src/core/guard.ts:68` (`status: GuardStatus`),
  `src/core/maintainer-check.ts:20` (`status: MaintainerCheckStatus`), and
  `start`'s top-level `status` (confirmed in the locked snapshot) all use
  `status`, while `src/core/check-gates.ts:34` (`overallStatus: GateStatus`),
  `src/core/doctor.ts:50` (`overallStatus: DoctorOverallStatus`),
  `src/core/release-check.ts:32`, `src/core/release-proof.ts:35`, and
  `src/core/ci-summary.ts:36` all use `overallStatus`. Confirmed in the
  locked snapshots too: `check-gates`'s shape has `"overallStatus": "string"`
  while `ready`, `guard`, and `start` have `"status": "string"`
  (`tests/contract/__snapshots__/json-shape.contract.test.ts.snap`). A
  generic client reading "the status" of any stable command has to know,
  per command, which of two field names to look at. **fix-before-1.0**.

- **"What to run next" has four different shapes** across commands that all
  express the same idea:
  - `nextAction: { command, reason }` ‚Äî `status`, `check-gates`, `ready`
    (all in the locked snapshot).
  - `nextCommand: { command, reason }` ‚Äî `start` (locked snapshot), a
    third name for the same pair.
  - flattened top-level `command` / `reason` (no wrapper object at all) ‚Äî
    `next` (`src/cli/commands/next.ts:72-91`, confirmed live: `next --json`
    keys are `['activeTask', ..., 'command', ..., 'reason', ...]` with no
    `nextAction` key and no `markdown` key, while `status --json` has both
    `nextAction` and `markdown`).
  - `nextActions: [{ command, reason }]` (plural, array) ‚Äî `guard`'s
    top-level field (locked snapshot), which also nests a *different*
    `nextActions` array inside `evidenceMap` for a related but distinct
    purpose.

  Three names (`nextAction`, `nextCommand`, bare fields) plus one
  singular/plural split, all locked into the frozen JSON contract as soon as
  Task 5's snapshots are treated as final. **fix-before-1.0**.

- **The JSON contract-lock only covers 8 of the ~30 `--json`-capable stable
  commands.** `JSON_COMMANDS` in `src/core/stable-surface.ts` is
  `['status', 'next', 'start', 'ready', 'guard', 'check-gates',
  'review-context', 'artifacts']`, and its own comment says "Start with the
  confirmed set below and extend during WS2." At least 24 more stable
  commands register `.option('--json', ...)` and are not in this list or in
  `json-shape.contract.test.ts` (`doctor`, `create-task`, `verify`,
  `summarize`, `handoff`, `runs`, `show-run`, `intent`, `maintainer-check`,
  `report`, `badge`, `upgrade-harness`, `ci-summary`, `release-notes`,
  `release-check`, `release-proof`, `npm-status`, `schemastore`, `github`,
  `policy`, `task`, `install-agent`, `list-templates`, `version`). Freezing
  the surface without a snapshot lock for most of it means regressions in
  those shapes won't be caught by `contract:check`. **fix-before-1.0**
  (expand `JSON_COMMANDS` and regenerate snapshots as part of the fix
  pass ‚Äî this is exactly what the comment already invites).
  **Resolved in Pass C:** `JSON_COMMANDS` now locks 36 of the 37
  `--json`-capable stable commands (command groups are represented by one or
  more read-only subcommands, e.g. `task list`, `policy status`, `context
  pack`, `loop status`); the one remaining gap (`show-run`) is documented
  with a reason in `KNOWN_NON_JSON` in
  `tests/contract/json-shape.contract.test.ts`, and a completeness test
  asserts every stable command is either locked, deferred, or confirmed to
  not register `--json` at all (`mcp-server`, `completion`). See
  `docs/1.0-consistency-audit.md`'s Resolution status section for the full
  coverage count.

## 3. Exit codes

The 0/1 model is applied **consistently**: every checker-style command
(`doctor`, `verify`, `check-gates`, `release-check`, `release-proof`,
`maintainer-check`, `ship`) sets `process.exitCode = 1` when its own
"fail" status is reached, and every generator-style command (`report`,
`badge`, `release-notes`, `ci-summary`, `summarize`/`handoff`, `artifacts`,
`prepare-pr`) never fails on content status, only on hard errors (bad output
path, bad option, etc.). That split is intentional and holds up across every
file checked (`grep -rn "process\.exitCode" src/cli/commands/*.ts`, 25
matches, all traced).

No command anywhere uses exit code `2`. The task brief's assumed
`0 = ok / 1 = failure / 2 = misuse` convention does not exist in this
codebase: commander's own default handling of unknown commands/options also
resolves to exit `1`, confirmed live:

```
$ node dist/cli/index.js badbadcommand;        # exit=1
$ node dist/cli/index.js status --nonexistent-flag  # exit=1
```

This is internally consistent (there is no command that uses `2` while
others use `1` for the same class of error), just narrower than the brief's
assumed three-way split. **accept-as-is** ‚Äî recommend a documentation
clarification in `docs/stability.md`'s exit-code axis (0 = success,
non-zero always means "1") so downstream consumers don't assume a `2` code
exists, but this is a docs nit, not a behavior fix.

Two related, non-breaking asymmetries worth naming but not blocking the
freeze:
- `guard` and `ready` only fail (`process.exitCode = 1`) when `--strict` is
  passed and status isn't `pass`/`ready` (`src/cli/commands/guard.ts:49-55`,
  `src/cli/commands/ready.ts:23-25`) ‚Äî advisory by default, matching their
  own `--help` text ("exit non-zero when ... is warn or fail" / "... is
  blocked"). **accept-as-is**, this is documented, deliberate behavior.
- `doctor` alone has a third flag, `--advisory`, that suppresses the
  default fail-on-`fail` exit code entirely (`src/cli/commands/doctor.ts:97`).
  No other checker command has an equivalent opt-out. **accept-as-is**
  (additive; doctor's embeddable-diagnostics use case plausibly needs this
  and no other command has asked for it yet).

## 4. Path redaction

Where `--redact-paths` exists and is wired to `redactLocalRoots`
(`src/core/redaction.ts:59`), behavior is uniform: every call site redacts
to the same default placeholder, `[git-root]`, and every call site is a
straight pass-through of `options.redactPaths === true`
(`grep -rn "redactLocalRoots(" src/core/*.ts src/cli/commands/*.ts`, 20
matches, all using the default replacement). Several commands
(`schemastore`, `github`, `runs`, `policy`, `artifacts`, `release-proof`,
`task list`) explicitly accept `--redact-paths` as a documented no-op
because their output structurally can't contain local paths ‚Äî this is a
deliberate, good pattern for uniform scripting ("always safe to pass
`--redact-paths`"), not an inconsistency.

Three real gaps:

- **`upgrade-harness --redact-paths` uses a different placeholder,
  `[agentloop-root]`, instead of the shared `[git-root]`.**
  `src/cli/commands/upgrade-harness.ts:11-16` hand-rolls
  `redactReportPaths()` instead of calling `redactLocalRoots`, replacing
  only `report.targetDirectory` with the literal string
  `'[agentloop-root]'`. Confirmed live: `upgrade-harness --help` shows
  `--redact-paths  replace the absolute AgentLoop root with
  [agentloop-root]`, the only one of ~25 `--redact-paths` registrations
  with different wording and a different output token. A script that
  greps redacted AgentLoopKit output for the standard `[git-root]` token
  (a reasonable thing to do, since it's the placeholder for every other
  command) will silently fail to detect that `upgrade-harness` redacted
  anything. **fix-before-1.0** ‚Äî route it through `redactLocalRoots` like
  everyone else, or document the deliberate divergence before it's frozen.

- **`agentloop init` leaks the absolute working directory and git root
  unconditionally, with no way to redact it.** `src/core/init.ts:306,340`
  set `targetDirectory: cwd` (the raw absolute `cwd`), and
  `src/core/init.ts:357` sets `git.root` from the resolved absolute git
  root. `src/cli/commands/init.ts` prints both directly in human output
  (`Target: ${result.targetDirectory}`, `Git root: ${result.git.root}`,
  lines 68/74) and in `--json` output (the whole `result` is dumped at line
  62), and registers no `--redact-paths` option at all. This is the first
  command most users/agents run, and its output (which plausibly ends up
  in CI logs, agent transcripts, or bug reports) always contains the local
  filesystem path including any username in the home directory. **fix-before-1.0**.

- **`agentloop create-task` leaks the absolute output path unconditionally,
  with no way to redact it.** `createTaskContractFile` in
  `src/core/task-contract.ts:262` returns `{ path: absolutePath, ... }` ‚Äî an
  absolute path, not repo-relative. `src/cli/commands/create-task.ts` prints
  it directly (`Task contract created: ${inlineCode(result.path)}`, line
  519; `activeTask.path` similarly; both also appear in the `--json` output
  at lines 504-517) and registers no `--redact-paths` option anywhere in the
  file. Every other command that writes a task-adjacent artifact (`report`,
  `badge`, `ci-summary`, `verify`, `release-notes`) supports redaction of
  its own output path. **fix-before-1.0**.

## 5. Newest surfaces (hardest look)

### `loop` and its subcommands

`src/cli/commands/loop.ts` defines six subcommands (`create`, `tick`, `run`,
`scorecard`, `status`, `report`). All six share one `printOutput()` helper
(line 33) and one `printJsonErrorIfNeeded()` helper (line 25), so their
`--json`/`--redact-paths`/error-envelope behavior is byte-for-byte identical
across subcommands ‚Äî this is the most internally consistent command group
in the whole surface. **accept-as-is**, with evidence.

One forward-looking design note, not a defect: `--cadence <cadence>`
currently accepts only the literal value `'manual'`
(`src/core/loop-contract.ts:32,758-760`, which throws
`LOOP_CADENCE_INVALID` for anything else). This is a deliberate placeholder
for a single-valued enum; adding new cadence values later is additive and
compatible with the freeze policy. **accept-as-is**.

### Baseframe integration (`create-task --from-projscan`, `check-gates --from-agentflight`)

- The `--task` semantic collision on `check-gates` (¬ß1) lives here ‚Äî the
  same flag name means "task contract path" everywhere else and "Baseframe
  task ID" only for this one AgentFlight-reconciliation path. This is the
  single highest-priority rename before the freeze, precisely because it's
  the newest, least-reviewed corner of the surface. **fix-before-1.0**
  (already counted in ¬ß1).
- `src/cli/commands/check-gates.ts:63-65,77` destructures both
  `fromAgentflight` and `fromAgentFlight` from options, then does
  `options.fromAgentflight ?? options.fromAgentFlight`. Only one flag,
  `--from-agentflight`, is ever registered (line 59), and commander's
  camelCase conversion only ever populates `fromAgentflight` (one lowercase
  word after the hyphen split) ‚Äî `fromAgentFlight` is dead code that can
  never be populated. Harmless today, but it's a small tell that this path
  hasn't had a close pass yet. **accept-as-is** (no behavior difference;
  worth a drive-by cleanup whenever the file is touched next, not worth a
  dedicated fix commit).
- `create-task --from-projscan`'s missing `--redact-paths` and absolute
  `task.path` leak (¬ß4) also live in this integration surface ‚Äî
  **fix-before-1.0** (already counted in ¬ß4).

Given the collision is a genuine rename and the redaction gap is a genuine
leak, but both fixes are small, isolated, and don't require reclassifying
the whole Baseframe integration as experimental, no
**defer-to-experimental** tag applies here ‚Äî the underlying reconciliation
logic (`evaluateAgentFlightResult`, gate promotion, exit-code handling) is
otherwise consistent with the rest of `check-gates`.

### `release-*` (`release-notes`, `release-check`, `release-proof`)

All three follow the shared checker/generator conventions well:
`release-check` and `release-proof` are checkers (`--strict`, exit 1 on
`overallStatus === 'fail'`, matching `doctor`/`check-gates` exactly);
`release-notes` is a generator (`--write`/`--out`, no status-driven exit
code, matching `report`/`badge`). `release-proof --redact-paths` is a
documented no-op ("release proof prints no local roots", line 199-201 of
`src/cli/commands/release-proof.ts`) consistent with the no-op convention
in ¬ß4. **accept-as-is**, this group is ready to freeze as designed. The
one placeholder-token defect found in this neighborhood
(`upgrade-harness`) is adjacent to, but not part of, the `release-*` family
itself ‚Äî it's listed under ¬ß4.

Two new findings surfaced empirically while probing fresh-repo `--json`
output for this pass (not fixed here ‚Äî test/lock-only pass):

- **`agentloop init --json --redact-paths` still leaks absolute paths.**
  Only `targetDirectory` and `git.root` are passed through `redact()` in
  `src/cli/commands/init.ts`; the `created`/`updated`/`skipped` arrays
  (spread from `...result`) are absolute paths and are not redacted at all,
  even with `--redact-paths` set. This is narrower than, and separate from,
  finding #7 (already fixed in Pass A), which only covered `targetDirectory`
  and `git.root`.
- **`release-proof --json` does not emit a JSON error envelope for an
  invalid npm package name.** `checkReleaseProof` calls `checkNpmStatus`
  (`src/core/release-proof.ts:577`), which calls
  `validateNpmPackageName(packageMetadata.name)`
  (`src/core/npm-status.ts:211`) unconditionally when the `npm` channel is
  checked, with no CLI-level override. If `package.json`'s `name` fails npm's
  naming rules (e.g. contains uppercase letters, as many generated/temp
  project directory names do), the thrown `AgentLoopError` is not caught by
  `release-proof`'s action (`src/cli/commands/release-proof.ts:256-269` only
  catches `ReleaseProofOnlyError`/`ReleaseProofTimeoutError`/
  `ReleaseProofCaptureError`), so it crashes to stderr as plain text with
  exit 1 regardless of `--json`. Same class of bug as already-fixed finding
  #2, in a different command.

The 9 accept-as-is findings are deliberately frozen as-is; a `docs/stability.md`
exit-code clarification (finding #12) may be folded into positioning work.
