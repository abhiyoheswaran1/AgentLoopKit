# Interview Cycle 4

## Context

AgentLoopKit `0.1.0` was published to npm. The CLI works, but the README relies on prose and tables. For a GitHub launch, the product needs faster visual comprehension without turning into a marketing site.

## Personas interviewed

- Indie Hacker Using Codex
- Open Source Maintainer
- Developer Experience Designer
- AI-Skeptical Senior Engineer
- Power User / Agentic Engineer

## Feedback summary

The strongest signal: the first viewport should prove what AgentLoopKit does before readers scroll into command tables. The panel rejected a hosted landing page and chose README-native visuals generated from local source files.

## Raw simulated feedback

### Indie Hacker Using Codex

- Liked: one-command install and local artifacts.
- Confused: what the generated harness looks like before installing.
- Needs before using it: a quick visual showing the files and loop.
- Would recommend/star it if: the README makes the workflow obvious in under a minute.
- Would abandon it if: setup looks like process overhead.

### Open Source Maintainer

- Liked: no telemetry, no postinstall, deterministic summaries.
- Confused: whether screenshots are decorative or generated from the tool.
- Needs before using it: transparent asset generation instructions.
- Would recommend/star it if: the README feels trustworthy and reproducible.
- Would abandon it if: images make the package look like an AI SaaS.

### Developer Experience Designer

- Liked: concrete CLI flow.
- Confused: the previous first viewport had no artifact preview.
- Needs before using it: screenshots that show task, verification, and handoff artifacts.
- Would recommend/star it if: the README communicates the workflow at a glance.
- Would abandon it if: visuals use generic dashboards or vague AI graphics.

### AI-Skeptical Senior Engineer

- Liked: no LLM dependency and no hidden network calls.
- Confused: whether "agentic engineering" means actual checks or just markdown.
- Needs before using it: verification evidence shown plainly.
- Would recommend/star it if: screenshots highlight tests, typecheck, build, and review evidence.
- Would abandon it if: README copy overpromises.

### Power User / Agentic Engineer

- Liked: multi-agent compatibility and task contracts.
- Confused: how `init`, `doctor`, `create-task`, and `summarize` connect.
- Needs before using it: a terminal demo.
- Would recommend/star it if: one GIF shows the loop from install to handoff.
- Would abandon it if: visual polish hides CLI details.

## Product council debate

- Abhi: Use visuals to strengthen the wedge, not to turn this into a landing page.
- Maya: Keep generated assets out of runtime code and avoid new CLI dependencies.
- Elias: README visuals help GitHub conversion, but the source files must be committed.
- Nora: Show the loop, artifacts, and terminal command flow. Skip abstract AI imagery.
- Samir: Absolute image URLs are safer for npm rendering than packaging extra runtime files.
- Lina: A GIF of the real CLI is more useful than a static mock.
- Tom: Verification screenshot matters more than brand polish.
- Rachel: A stronger README helps small teams understand the review discipline.

## Decision

Create README assets with Playwright and VHS, embed them in README, bump a patch release, and add a workflow guard so already-published versions do not fail a release-triggered publish job.

## Non-decisions

- No hosted docs site.
- No product dashboard.
- No telemetry or analytics.
- No screenshots generated by an AI image model.

## Resulting tasks

- Add `docs/assets/readme/agentloopkit-cli.tape`.
- Generate two PNG screenshots with Playwright.

- Update README and changelog.

## Success criteria

- README has visual proof above the fold.
- Assets can be regenerated locally.
- Package checks still pass.
- `npx agentloopkit version` works from npm.
- Follow-up release ships without duplicate npm publish failures.
